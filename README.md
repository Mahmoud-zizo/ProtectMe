# Pro Sign - 3M Authorized Center

## 🚀 Overview

**Pro Sign** is a modern, full-stack web application built for a premier 3M Authorized auto detailing and car care center. It acts as a centralized digital platform connecting customers with the service center. Users can seamlessly browse services, request custom quotes, and book appointments, while providing administrators with a powerful, comprehensive dashboard to manage requests, branches, and generate pricing quotes.

## 🛠️ Technology Stack & Dependencies

Built with cutting-edge tools to ensure high performance, robust security, and an exceptional user experience:

- **Framework**: Next.js 16 (App Router)
- **Frontend Library**: React 19
- **Styling**: Tailwind CSS v4, Shadcn UI, Radix UI Primitives
- **Animations**: Framer Motion (`motion`), `tw-animate-css`
- **Authentication**: Auth.js (NextAuth v5 Beta)
- **Database**: PostgreSQL
- **ORM**: Prisma Client (with `@prisma/adapter-pg` for Auth.js)
- **Language**: TypeScript throughout the entire stack

## ✨ Core Features

### For Users 🚗

- **Secure Authentication**: Passwordless or password-based login via Credentials, alongside Google OAuth integration.
- **Service Browsing**: Explore available detailing, protection, and tinting services through an interactive, animated UI.
- **Request Booking**: Submit detailed service requests, including snapshots of customer data, car information (Make, Model, Year), preferred branch, and selected services.
- **Quote Management**: Receive and review custom quotes generated by administrators based on initial requests.
- **Status Tracking**: Keep an eye on the request lifecycle status, from initial booking to completion.

### For Administrators 🛡️

- **Role-Based Access**: A strictly protected admin panel accessible exclusively to users carrying the `ADMIN` role.
- **Request Management**: Comprehensive dashboard to review incoming customer requests and inspect car details.
- **Quote Generation**: The ability to evaluate a request and issue a custom `Quote`, specifying exact pricing, expected duration in days, and additional administrative notes.
- **Entity Management**: Control core business objects like Branches, Services, and track Available Slots.
- **Workflow Control**: Update and mutate request states across its lifecycle (Pending, Quoted, Confirmed, Completed, Cancelled).

## 🔄 Core Workflow, Logic, & Behind the Scenes

The application revolves around a state-machine-inspired **Request System**. Deeply integrated with Prisma, the flow is designed to mimic a real-world service center interaction securely:

1. **Submission (`PENDING_REVIEW`)**: A User submits a `Request`. This creates a relational cluster in the database bridging the user's ID, the chosen `Branch`, and populates `RequestService` with the requested jobs. The request is assigned a `PENDING_REVIEW` state.
2. **Review & Quote (`QUOTED`)**: An Admin accesses the admin dashboard, inspects the `PENDING_REVIEW` requests, and issues a `Quote` (which dictates the total price and expected duration). Issuing the quote automatically shifts the request state to `QUOTED`.
3. **Approval (`CONFIRMED`)**: The User is notified in their dashboard. Upon reviewing the quote, they can accept the terms. The system locks the price and changes the status to `CONFIRMED`.
4. **Fulfillment (`COMPLETED` or `CANCELLED`)**: Once the physical service is rendered at the selected branch, the Admin finalizes the workflow, marking the record as `COMPLETED`.

## 🗄️ Database Architecture & Schemas

Governed by **Prisma** on PostgreSQL, the database is highly relational and completely typed:

- **`User` / `Account`**: Managed by NextAuth. Handles roles (`USER`, `ADMIN`) and OAuth account links.
- **`Service`**: The active catalog of offerings (e.g., "Nano Ceramic", "Window Tint"), complete with descriptions and base prices.
- **`Branch` & `AvailableSlot`**: Represents physical locations and their calendar booking availability.
- **`Request`**: The central pivot table linking passing relational data to snapshots of customer/car details. Employs cascading relations.
- **`RequestService`**: A many-to-many junction table resolving Requests to multiple requested Services.
- **`Quote`**: Admin-generated pricing object tied `1:1` with a Request.

## 🏗️ Getting Started

### Prerequisites

Ensure you have the following installed on your machine:

- Node.js (v18.17+ recommended)
- PostgreSQL (Database cluster up and running)

### Setup Instructions

1. **Clone the source code**

   ```bash
   git clone <repository-url>
   cd auth-test
   ```

2. **Install project dependencies**

   ```bash
   npm install
   ```

3. **Configure the Environment**
   Create a `.env` file in the root directory. Configure your connection strings and auth secrets:

   ```env
   # Database (PostgreSQL)
   DATABASE_URL="postgresql://user:password@localhost:5432/prosign"

   # NextAuth
   AUTH_SECRET="your-nextauth-secret-key" # Run `npx auth secret` to generate
   AUTH_GOOGLE_ID="your-google-oauth-client-id"
   AUTH_GOOGLE_SECRET="your-google-oauth-client-secret"
   ```

4. **Initialize Database Models**
   Synchronize your PostgreSQL database with the Prisma schema and generate the client:

   ```bash
   npx prisma db push
   npx prisma generate
   ```

5. **Start up the application**
   ```bash
   npm run dev
   ```
   The application will boot up at `http://localhost:3000`.

## 🤖 AI / LLM Developer Guide

If you are an AI assistant or LLM, please follow these architectural patterns and rules when modifying this codebase:

### 1. **Next.js App Router Paradigms**

- The project strictly uses the Next.js **App Router** (`app/` directory).
- **Server Components by Default**: All components inside `app/` are React Server Components (RSC) unless explicitly marked. Do not add `'use client'` unless the component strictly requires React Hooks (`useState`, `useEffect`), browser APIs, or client-side interactivity (e.g., `framer-motion`).
- **Data Fetching & Mutations**: Avoid raw `fetch()` on the client for mutations. Prefer explicitly defined **Server Actions** placed in `app/actions/`. Handle complex data retrieval inside the RSC page (`page.tsx`) before passing it down as props.

- Standard instances of Prisma client should be imported via a shared singleton at `lib/prisma.ts` or derived exactly as established in the workspace.
- **Custom Output Catch**: The Prisma client explicitly generates outputs to `../generated/prisma` as specified in `prisma/schema.prisma` rather than standard `@prisma/client`. Pay attention to these explicit import paths natively resolving in the application.
- **Migration & Schema**: If you add models to `prisma/schema.prisma`, you must prompt the user to run `npx prisma db push` and `npx prisma generate` for the application to register the typed types.

### 4. **Styling and Components**

- **Tailwind CSS v4 framework** is in place.
- **Utility Method**: Use the `cn()` function available in `lib/utils.ts` (a wrapper over `clsx` and `tailwind-merge`) when adding conditional classes or exposing `className` props.
- **UI Components**: Use the established `components/` primitive architecture (leveraging **Radix UI** and **Shadcn UI**). Do not reinvent primitives (Buttons, Inputs, Dialogs) if an equivalent Shadcn component exists.
- **Animations**: Prefer `framer-motion` for complex keyframes. Limit the arbitrary use of overly complex standard CSS animations unless handled by `tw-animate-css`.
